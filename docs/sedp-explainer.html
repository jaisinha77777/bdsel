01 · Basics

The problem, in plain terms

Kafka is a system for moving streams of data. To handle a lot of traffic at once, a stream ("topic") is split into partitions — independent lanes that can be processed in parallel. Think of a supermarket: one long checkout queue is slow, so the store opens several lanes and splits shoppers across them.

The problem is that shoppers don't distribute themselves evenly. If one lane happens to get all the big carts, it backs up — a queue builds even though other lanes sit idle. In Kafka terms this is data skew, and the backed-up queue is consumer lag: how far behind a partition's reader is from the data that's already arrived.

Reactive (traditional)

Wait until a lane is visibly jammed — lag is already high, customers are already waiting — then open a new lane. By definition, the damage already happened before anyone acted.

Predictive (SEDP)

Watch each lane's trend, forecast which one is about to get slammed, and act — split it, merge two quiet ones, or move a lane to a less busy cashier — before the queue actually forms.

Everything below is the machinery that makes "forecast, then act early" precise enough to automate: how to predict a lane's near-future load, how to turn that forecast into a split/merge/reassign decision, and how to check, with real numbers, that any of this is actually working.

02 · Basics

System anatomy

Five components hand off to each other in a loop, once per monitoring cycle (every few seconds):

1 · input
Load Monitor
Keeps a rolling window of each partition's recent throughput, lag and latency.
2 · forecast
Predictor
One of six algorithms turns "recent history" into "next value."
3 · score
PPEA
Combines the forecast with live lag/latency into one Pressure Score per partition, and decides.
4 · act
Evolution Engine
Executes the decision: split a hot partition, merge two idle ones, or reassign one to another broker.
5 · learn
Feedback Controller
Checks a few cycles later whether the action actually helped, and nudges the scoring weights if not.
the output of step 5 feeds back into step 3's weights — the "self-evolving" part
03 · Core mechanism

The core decision: Partition Pressure Score

Every cycle, each partition gets reduced to a single number — how "stressed" it is — called the Partition Pressure Score, or PPS. It's a weighted sum of three signals:

PPS(p) = w1·L(p) + w2·G(p) + w3·T(p) L(p) predicted load — this cycle's forecast (§4) G(p) consumer lag — how far behind the reader already is T(p) processing latency — how long each record takes to handle w1+w2+w3 = 1 (weights are normalized so PPS stays on one interpretable scale)

The weights start at w₁=0.5, w₂=0.3, w₃=0.2 — the forecast matters most, because it's the only signal that's early; lag and latency are already-happened symptoms.

Why normalize the weights to sum to 1?

A weighted sum where the weights sum to 1 is a convex combination — the result is guaranteed to land inside the range spanned by the three inputs, rather than blowing up or shrinking depending on how many terms you added. It also makes the weights directly readable as "percent of the decision each signal accounts for."

Turning a score into an action

ActionRuleReading
SplitPPS(p) > split_thresholdThis lane is overloaded — open a second one.
MergePPS(p₁) & PPS(p₂) < merge_thresholdTwo neighboring lanes are both quiet — combine them.
Reassignσ / μ > reassign_thresholdLoad is unevenly spread across all lanes — move the hottest one to a less busy machine.
What is σ / μ ?

The standard deviation of all partitions' scores divided by their mean — a standard statistics measure called the coefficient of variation. It answers "how spread out is the load, relative to its own average?" — which is more meaningful than raw spread alone, since a spread of 500 means something very different when the average load is 1,000 versus 1,000,000.

The "self-evolving" part: adaptive weights

Every action is logged, and a few cycles later the Feedback Controller checks: did lag actually drop by at least 10%? If not, the weight of whichever signal looked most responsible for the bad call gets nudged up, and everything is renormalized back to summing to 1. Over time the three weights drift toward whatever mix has actually been working on the traffic this deployment sees.

04 · The mathematics

Six ways to predict the future

All six algorithms answer the same question — "given the load so far, what's the next value?" — with the same shape of answer: feed in one new number, get one prediction out. They differ entirely in how much history they trust, and how.

4.1 · Exponentially Weighted Moving Average (EWMA)

smoothing
Intuition

A running average that never forgets the past entirely, but cares less and less about it the further back it goes. Each new point nudges the average toward itself by a fraction α.

St = α·xt + (1−α)·St−1  // S = smoothed estimate, α ∈ (0,1)

Unroll the recursion one step at a time and a pattern appears — St is secretly a weighted average of every past observation, with the weights shrinking geometrically the further back you go:

St = α·xt + α(1−α)·xt−1 + α(1−α)²·xt−2 + …

That's the whole trick: α close to 1 means "trust the newest point almost entirely" (reacts fast, noisy); close to 0 means "barely update" (smooth, slow to react). SEDP uses α = 0.3 by default.

+ Trend — this repo's original estimator

EWMA alone predicts "next = same as the smoothed level," which always lags behind a rising or falling trend. The fix used here: extrapolate using the smoothed series' own most recent slope.

trendt = St − St−1 predicted = St + trendt
Where this is weaker than it looks

trend is just the raw first difference of an already-smoothed series — it is not itself smoothed, so it inherits more noise than a trend estimate that gets its own dedicated smoothing (see Holt's method, next). That's a real, structural difference, not a bug.

4.2 · Holt's Linear Method

smoothing (double exponential)
Intuition

Run two EWMAs at once — one tracking the current level, one tracking the current slope — each with its own forgetfulness dial.

levelt = α·xt + (1−α)·(levelt−1 + trendt−1) trendt = β·(levelt − levelt−1) + (1−β)·trendt−1 predicted = levelt + trendt

The trend line now has its own memory (β = 0.1 by default, slower to change than the level's α = 0.3) instead of being recomputed from scratch every step — which is exactly what fixes EWMA+Trend's noise problem above.

Verified

Fed a perfectly straight, noiseless ramp — after the first couple of steps, Holt's prediction error drops to exactly 0.0. That's the textbook property of double exponential smoothing on a linear trend, confirmed directly against this implementation.

4.3 · Autoregressive AR(3)

classical time series
Intuition

Assume the next value is some fixed linear combination of the last p values — "3 parts of what happened 1 step ago, minus 1 part of 2 steps ago, plus…" — and let the data itself tell you those parts.

yt = φ1yt−1 + φ2yt−2 + φ3yt−3 + εt

Finding the coefficients φ₁, φ₂, φ₃ is a classic ordinary least squares problem: pick the φ's that minimize total squared prediction error over the recent window. That reduces to solving a small linear system (the normal equations):

XTX φ = XTy

where each row of X is one lag-window [yt−1, yt−2, yt−3]. The window is first centered (its own mean subtracted off) so the system stays numerically well-behaved even when raw load values are in the hundreds of thousands. The system itself is solved with Gaussian elimination — the standard hand (and computer) method for solving several linear equations at once by systematically cancelling variables row by row.

Verified

This project's hand-written Gaussian elimination solver was run against 2,000 random test systems and compared to numpy.linalg.solve — maximum disagreement: 2.2 × 10⁻¹⁶, i.e. floating-point-exact.

Where AR(3) struggles

A fixed linear combination of 3 past points is a poor fit for a strong, steady ramp — it can only fake a trend by pushing its coefficients toward summing to ~1 (a "unit root"), which is a fragile, indirect way to represent something Holt's method represents directly. This is exactly why AR(3) ranks last on this project's trending workload — theory and measurement agree.

4.4 · Kalman Filter (constant velocity)

state-space / Bayesian
Intuition — "the GPS algorithm"

Your phone's GPS chip is noisy, but you know objects move continuously — they don't teleport. The Kalman filter keeps a running belief about a hidden true state (here: position and velocity of the load), predicts forward using physics (next position = position + velocity), then corrects that prediction using the new noisy measurement — trusting whichever of the two (its own physics-based guess, or the new measurement) it currently has more confidence in.

The belief isn't just a number — it's a number plus a confidence, tracked as a 2×2 covariance matrix P. Each cycle runs two steps:

// 1. predict — physics only, no new data yet x̂ ← F·x̂ // position += velocity P ← F·P·FT + Q // uncertainty grows without new info // 2. update — blend the prediction with the new measurement z K = P·HT / (H·P·HT + R) // Kalman gain: how much to trust z x̂ ← x̂ + K·(z − H·x̂) // nudge belief toward the measurement P ← (I − K·H)·P // confidence improves after a good look
What the Kalman gain K is actually doing

K is a self-adjusting dial between 0 and 1. If the sensor is very noisy relative to how confident the model is (R large), K shrinks and the filter mostly trusts its own physics-based prediction. If the model's own uncertainty P is large — it hasn't seen data in a while, or the regime just changed — K grows and the filter leans on the new measurement instead. Unlike EWMA's fixed α, this trade-off recomputes itself every single cycle.

Verified

This implementation's predictions were compared step-by-step against an independent textbook reimplementation of the same constant-velocity filter over 201 steps — maximum disagreement: 2.9 × 10⁻¹¹.

4.5 · Linear Regression (sliding window)

regression
Intuition

The simplest of the six: look at the last n points, draw the best-fit straight line through them, and read off one step past the end of that line.

slope = Σ(i−ī)(yi−ȳ) / Σ(i−ī)² // closed-form least squares intercept = ȳ − slope·ī predicted = intercept + slope·n

Because the window is small and fixed (6 points by default), it reacts quickly to recent change but has no memory of anything older — it's the most "local" of the six forecasters.

Verified

Matched numpy.polyfit's line-fit-and-extrapolate output exactly on a synthetic noisy ramp.

4.6 · Game-Theoretic Ensemble

game theory / online learning — this repo's own contribution
Intuition — a repeated game against your own five experts

Treat the five algorithms above as competing "experts" playing a repeated game. Every round, all five make a prediction; once the true value shows up, whoever was wrong loses influence, and whoever was right gains it — automatically, just by renormalizing. The final forecast is a weighted blend, not a single winner-take-all pick.

The weight update is called Multiplicative Weights (also: Hedge):

lossi = |predictioni − actual| / (worst error this round) // rescaled to [0,1] wi ← wi · e−η·lossi // punish bad experts exponentially w ← w / Σw // renormalize back to a probability forecast = Σ wi·predictioni

The "no-regret" guarantee, in plain terms

Multiplicative Weights comes with a mathematical promise: over any stretch of T rounds, your total error is guaranteed to stay within O(√(T·log N)) of the error the single best expert would have had, if you'd known in advance which one that was — for N experts. In plain terms: you never do much worse than hindsight's best single choice, without ever needing hindsight.

This implementation adds Fixed-Share: after every weight update, a small slice of weight (5%) is leaked back to a uniform 1/5 across all experts. Plain Multiplicative Weights eventually locks onto one expert and stops tracking anyone else — Fixed-Share keeps a standing bet on the others, so when the traffic pattern suddenly shifts (calm → burst), a previously-losing expert can reclaim influence within a few rounds instead of never.

A genuine caveat, not just a footnote

The textbook O(√(T·log N)) bound assumes each round's loss is measured on a fixed, known scale. Here, losses are rescaled every round by that round's own worst error — necessary, since forecast error has no fixed a-priori range — but it means the strict textbook proof doesn't transfer unmodified. It's best understood as a well-motivated adaptive variant of Hedge, not a literally proven bound on this exact code.

Verified

Best raw accuracy of all six algorithms on this project's workload, and the only one with positive skill (beats a naive forecast) at 320,000 points. The literal cost: it runs all five other experts internally every step, making it ~90× slower than the fastest single algorithm — a real trade-off, not a flaw, and the reason the composite score in §7 treats speed as a first-class input rather than an afterthought.

05 · Measuring quality

Is a forecast any good? — the metrics

Every algorithm above outputs one number per cycle. To compare six different algorithms fairly, that stream of predictions has to be reduced to a handful of scores. Here's what each one actually means:

MetricFormulaReads as
MAEmean(|predicted − actual|)Average miss, in the same units as the data. Simple, robust to outliers.
RMSE√mean((predicted − actual)²)Like MAE but squares errors first — punishes big misses much harder than small ones.
MAPEmean(|error| / actual) × 100Average miss as a percentage. Unstable when actual is near zero.
R²1 − SSE / SSTFraction of the data's own variance the forecast explains, vs. just guessing the average every time.
Directional accuracy% of steps where sign(Δpredicted) = sign(Δactual)Did it call "up" vs "down" correctly? 50% = coin flip.
Skill(1 − MAE / naiveMAE) × 100% better than the dead-simple baseline "next = current value." Positive = actually earning its keep.
Theil's U2RMSE / naiveRMSESame idea as skill, using RMSE. <1 beats the naive baseline.
A verified weak spot: R² nearly doesn't discriminate here

R² compares a forecast against the weakest possible baseline — "always guess the overall average." On a workload with a strong ramp/burst pattern, that baseline is so bad that almost any reasonable forecaster beats it by a wide margin, so R² saturates near 1.0 for all six algorithms regardless of real quality differences between them.

Measured directly on this project's own 6-algorithm run: R² ranged 0.971–0.988 — a 1.7% spread — while skill ranged over 33 points on the exact same runs. R² also turned out to be 94.5% correlated with skill and 99.7% correlated with Theil's U2: three different-looking numbers carrying almost the same one signal. This is exactly why R² was dropped from the composite score in §7.

06 · Statistical rigor

Monte Carlo: why one run isn't enough

Every metric above is computed on one randomly generated workload. Run it again with a different random seed and the numbers shift — some algorithms get lucky on one seed and unlucky on the next. Reporting a single run risks mistaking noise for a real difference.

The fix is standard Monte Carlo evaluation: run the same comparison across several seeds (5, by default) and report both the mean and the standard deviation of each metric:

skill_mean = average(skill across seeds) skill_std = how much skill wobbles from one seed to the next

A low skill_std means an algorithm's quality is dependable; a high one means its good average is partly luck. This distinction becomes a real ingredient in the composite score next.

07 · Ranking algorithms

The composite score, built from scratch

Six algorithms, seven-plus metrics each, across Monte Carlo seeds — eventually you want one number to rank them by. Building that number well turns out to be its own small piece of math.

Step 1 — pick signals that don't just repeat each other

As shown in §5, MAPE / R² / Theil's U2 / skill are all different arithmetic views of essentially the same "error vs. a baseline" idea — stacking four of them into one score mostly just counts that one idea four times. The composite instead uses exactly three genuinely different signals:

  • accuracy — how good the forecast is
  • directional — does it get the direction of change right (a structurally different question from magnitude)
  • efficiency — how fast it runs, since a forecast that's too slow to compute isn't usable regardless of accuracy

Step 2 — put every signal on a scale that means the same thing every time

Skill is a percentage, DirAcc is a percentage on a different baseline, and throughput is a raw count spanning six orders of magnitude (10,000 to 2,000,000+ points/sec). Comparing them directly, or normalizing them relative to whichever algorithms happen to be selected in a dropdown, makes the score unstable — literally the same algorithm can score differently depending on who else is on screen. Every component here instead maps to a fixed 0–100 scale against a fixed reference point:

accuracy = clip( (risk_adj_skill − (−40)) / (20 − (−40)) × 100, 0, 100 ) // 0% skill (ties the naive baseline) → 50pt · +20% skill → 100pt · −40% → 0pt directional = clip( (DirAcc − 50) / (100 − 50) × 100, 0, 100 ) // 50% (coin flip) → 0pt · 100% (perfect) → 100pt efficiency = clip( (log₁₀(throughput) − log₁₀(10,000)) / (log₁₀(1,000,000) − log₁₀(10,000)) × 100, 0, 100 ) // log scale, because throughput spans orders of magnitude, not a linear range
What "risk-adjusted skill" means

risk_adj_skill = skill_mean − 1 × skill_std — an algorithm's Monte Carlo average skill, penalized by how much it wobbles between seeds. Two algorithms with the same average skill don't score the same if one of them is only good when it gets lucky; this term makes consistency count, not just the average.

Step 3 — combine

composite = 0.5·accuracy + 0.2·directional + 0.3·efficiency

Accuracy dominates (0.5), directional gets a real but smaller vote (0.2), and efficiency is weighted high enough (0.3) to matter — because on this project's own large-scale benchmark, the most accurate algorithm (the game-theoretic ensemble, §4.6) is also ~90× slower than the fastest, and a score that ignored that entirely would be quietly recommending the most expensive option every time.

Why this replaced an earlier version

An earlier composite blended five metrics using min-max normalization — scaling each metric relative to whichever algorithms were currently selected for comparison. Measured directly: the exact same algorithm, with byte-identical accuracy, scored 61.0 when compared against all six algorithms and 35.0 when compared against a two-algorithm subset — a 26-point swing with nothing about the algorithm itself changing.

The fixed-anchor version above was tested the same way: the same algorithm scored identically across four repeated trials against different comparison sets — the instability is gone by construction, not by luck.
